# Cursor Web Design Guide

이 문서는 Cursor 에이전트가 웹페이지, 랜딩 페이지, 브랜드 사이트, 제품 소개 페이지, 웹 앱 화면을 만들 때 따르는 **디자인 실행 가이드**다.

목표:

- 결과물이 흔한 AI 템플릿처럼 보이지 않을 것
- 디자인 방향이 분명할 것
- 레이아웃이 브랜드와 콘텐츠 맥락을 반영할 것
- 실제 구현 가능한 코드로 이어질 것
- 반응형과 접근성이 기본 이상일 것

이 문서는 컴포넌트 백과사전이 아니다.  
이 문서는 에이전트가 **무엇을 우선 판단하고, 무엇을 피해야 하는지**를 정하는 문서다.

---

## 1. 기본 태도

에이전트는 화면을 만들 때 아래를 기본값으로 삼는다.

1. 먼저 사용자의 목적과 맥락을 해석한다.
2. 그에 맞는 강한 미학 방향을 하나 고른다.
3. 레이아웃, 타이포, 컬러, 모션을 그 방향에 맞춰 일관되게 설계한다.
4. 흔한 SaaS 템플릿 조합을 자동 반복하지 않는다.
5. 설명만 하지 말고 실제로 동작 가능한 결과물을 만든다.

좋은 결과물은 “그럴듯한 섹션 나열”이 아니라, **의도가 느껴지는 화면**이다.

---

## 2. 가장 중요한 원칙

### 2.1 하나의 강한 방향을 먼저 정한다

디자인을 시작하기 전에 스스로 아래를 정리한다.

- 이 페이지는 누구를 위한 것인가
- 사용자가 첫 5초 안에 무엇을 느껴야 하는가
- 이 브랜드/서비스의 태도는 무엇인가
- 무엇이 이 화면을 기억하게 만드는가

디자인은 “예쁘게”가 아니라 **명확한 태도**를 가져야 한다.

예시 방향:

- 조용한 프리미엄
- 에디토리얼 매거진
- 실험적인 미래지향
- 따뜻한 공예 감성
- 공업적이고 기능적인 미학
- 시네마틱 다크 무드
- 장난감 같은 인터랙티브 감성

### 2.2 템플릿처럼 쌓지 않는다

아무 맥락 없이 아래 순서를 반복하지 않는다.

- Hero
- Features 3개
- Testimonials
- FAQ
- CTA
- Footer

이 구조는 편하지만 쉽게 진부해진다.  
항상 콘텐츠 우선순위와 브랜드 메시지에 맞춰 **서사를 다시 설계**한다.

### 2.3 디테일은 컨셉을 강화해야 한다

좋은 디자인은 장식이 많아서가 아니라, 모든 선택이 같은 방향을 향하기 때문에 좋아 보인다.

- 폰트는 분위기를 만든다
- 간격은 브랜드의 태도를 만든다
- 모서리와 선은 성격을 만든다
- 움직임은 제품의 자신감을 만든다

아무 이유 없는 그라디언트, 유리 효과, 네온, 과한 블러는 금지다.

### 2.4 구현 가능한 결과물을 만든다

결과물은 실제 프론트엔드 코드로 이어질 수 있어야 한다.

- CSS로 자연스럽게 구현 가능해야 한다
- 인터랙션은 멋보다 유지보수 가능성을 함께 고려한다
- 시각 효과는 브랜드를 살리는 수준에서만 정교하게 쓴다
- 랜딩 페이지는 스크롤 경험이 자연스러워야 한다
- 앱 화면은 정보 구조와 조작 흐름이 우선이다

---

## 3. 작업 순서

사용자 요청이 모호하더라도 아래 순서로 판단한다.

### 3.1 먼저 정리할 것

- 페이지 유형: 랜딩 / 브랜드 소개 / 상세 페이지 / 대시보드 / 폼 / 커머스 / 앱 화면
- 핵심 사용자
- 가장 중요한 행동: 구매 / 문의 / 가입 / 탐색 / 신뢰 형성 / 정보 이해
- 원하는 톤: 차갑게 / 따뜻하게 / 실험적으로 / 미니멀하게 / 권위 있게 / 부드럽게

### 3.2 다음으로 고를 것

- 레이아웃 시스템
- 타이포그래피 조합
- 컬러 시스템
- 인터랙션 밀도
- 시각적 포인트 한 가지

### 3.3 마지막으로 구현할 것

- 섹션 구조
- 컴포넌트 디테일
- 반응형 규칙
- 모션
- 접근성 처리

---

## 4. 추천 미학 방향

사용자 요청이 구체적이지 않다면, 아래 중 하나를 **명시적으로 선택하고 끝까지 유지**한다.

### 4.1 Quiet Premium

- 넉넉한 여백
- 절제된 컬러 수
- 고급스러운 serif + 정제된 sans 조합
- 큰 제목, 적은 장식
- 미세한 선과 섬세한 대비

좋은 사용처:

- 파인다이닝
- 뷰티
- 스튜디오
- 하이엔드 브랜드

### 4.2 Editorial Narrative

- 매거진 같은 비대칭 레이아웃
- 강한 제목 위계
- 이미지와 텍스트의 리듬감
- 격자를 일부 깨는 배치
- 과감한 크롭과 큰 여백

좋은 사용처:

- 브랜드 스토리
- 캠페인 페이지
- 패션
- 문화/콘텐츠

### 4.3 Futuristic Product

- 어두운 배경 또는 강한 대비
- 제한적인 발광 포인트
- 인터페이스처럼 정돈된 구조
- 선명한 하이라이트 컬러
- 빠르고 또렷한 인터랙션

좋은 사용처:

- AI
- 개발자 도구
- 생산성 앱
- 고기능 SaaS

### 4.4 Warm Crafted

- 자연스러운 색
- 유기적인 간격
- 부드러운 모서리
- 손맛이 느껴지는 텍스처
- 친근한 카피와 상호작용

좋은 사용처:

- F&B
- 로컬 브랜드
- 라이프스타일
- 크리에이터 제품

### 4.5 Playful Motion

- 움직임이 브랜드 경험의 핵심
- 구조는 단순하지만 반응은 기억에 남음
- 인터랙션 자체가 인상 포인트
- 사용성은 절대 희생하지 않음

좋은 사용처:

- 이벤트
- 프로모션
- 소비자 앱
- 실험적 포트폴리오

---

## 5. 페이지 타입별 전략

### 5.1 랜딩 페이지

랜딩 페이지는 정보 나열이 아니라 **확신을 쌓는 흐름**이어야 한다.

권장 흐름:

1. 첫 화면에서 한 줄로 가치 제안이 이해된다
2. 바로 아래에서 신뢰 또는 차별점이 보인다
3. 중간 구간에서 구체적인 사용 장면을 보여준다
4. 후반부에서 망설임을 제거한다
5. 마지막 CTA는 앞 내용을 요약하는 느낌이어야 한다

### 5.2 브랜드 소개 페이지

- 기능보다 무드와 관점이 중요하다
- 이야기의 순서를 설계한다
- 이미지, 짧은 카피, 레이아웃 리듬으로 몰입을 만든다
- 모든 섹션이 같은 카드 구조를 반복하지 않게 한다

### 5.3 프로덕트 상세 페이지

- 핵심 정보는 위로
- 시선 흐름은 요약 -> 근거 -> 세부 -> 행동
- 이미지와 텍스트는 서로를 보완해야 한다
- CTA는 스크롤 중에도 길을 잃지 않게 배치한다

### 5.4 대시보드 / 앱 화면

- 미학보다 정보 구조가 우선
- 카드 수를 늘리기보다 위계를 명확히 한다
- 중요한 수치, 상태, 작업을 먼저 노출한다
- 시각 효과보다 읽기 속도와 조작 안정성을 높인다

---

## 6. 섹션 설계 규칙

섹션은 “이름 있는 블록”이 아니라, 페이지 목적을 수행하는 **장면**이다.

각 섹션을 만들기 전 확인할 것:

- 이 섹션의 목적은 무엇인가
- 이전 섹션과 어떤 긴장감 차이를 만드는가
- 다음 섹션으로 왜 넘어가게 되는가
- 이 섹션만의 시각적 논리가 있는가

좋은 섹션의 특징:

- 제목만 읽어도 흐름이 이어진다
- 콘텐츠에 따라 레이아웃이 달라진다
- 같은 카드/그리드가 반복되지 않는다
- 중간중간 시각적 호흡 조절이 있다

나쁜 섹션의 특징:

- 카드 3개 반복
- 아이콘 3개 반복
- 배경색만 바꾼 유사 박스 반복
- 제목/설명/버튼 구조가 모든 섹션에서 동일

---

## 7. 타이포그래피 규칙

### 7.1 폰트 선택

기본적으로 **표정 있는 display 폰트 + 읽기 좋은 body 폰트** 조합을 우선 고려한다.

피해야 할 것:

- 무지성 `Inter`
- 근거 없는 `Space Grotesk`
- 아무 컨셉 없는 시스템 폰트 고정
- 성격이 충돌하는 폰트 3개 이상 혼용

좋은 기준:

- 브랜드 톤과 맞는가
- 숫자와 대문자가 예쁜가
- 긴 본문 가독성이 유지되는가
- 모바일에서도 무너지지 않는가

### 7.2 위계

타이포 위계는 분명해야 한다.

- Hero headline은 과감하게
- 섹션 제목은 구조를 읽게
- 본문은 빠르게 이해되게
- 캡션과 메타 정보는 과하지 않게

좋은 결과물은 크기만 다른 것이 아니라,

- 자간
- 행간
- 굵기
- 대소문자 사용
- 정렬 방식

까지 함께 설계되어 있다.

---

## 8. 컬러 시스템 규칙

컬러는 적을수록 좋다는 뜻이 아니라, **역할이 명확해야 한다**는 뜻이다.

먼저 아래 역할을 정의한다.

- 배경 기본색
- 배경 보조색
- 텍스트 기본색
- 텍스트 보조색
- 포인트 색
- 경계선 색
- 상태 색

권장:

- CSS 변수로 관리
- 포인트 컬러는 한두 개로 제한
- 중간톤으로 깊이를 만든다

금지:

- 흰 배경 + 보라 그라디언트의 상투적 조합
- 모든 요소에 강한 색 사용
- 대비가 약해 읽기 어려운 저채도 디자인
- 다크 모드에서 회색만 많은 흐린 화면

---

## 9. 공간과 레이아웃 규칙

레이아웃의 수준은 무엇을 넣었는가보다 **어떻게 비웠는가**에서 갈린다.

반드시 지킬 것:

- 간격 스케일을 정하고 반복한다
- 컨테이너 폭에 논리가 있어야 한다
- 섹션 간 간격은 컴포넌트 내부 간격보다 분명히 커야 한다
- 텍스트 폭은 너무 길지 않게 제어한다
- 카드 간 거리보다 큰 블록 간 관계를 먼저 본다

레이아웃 포인트:

- 완벽한 대칭만 고집하지 않는다
- 필요한 곳에는 비대칭과 오프셋을 쓴다
- 중요한 순간에는 풀블리드 요소를 써도 된다
- 지나친 복잡성 없이 단조로움을 깨야 한다

---

## 10. 컴포넌트 사용 원칙

컴포넌트는 쌓는 것이 아니라 **문맥에 맞게 선택**해야 한다.

### 10.1 버튼

- 주요 CTA는 정말 중요할 때만 강하게
- 보조 버튼까지 같은 무게를 가지면 안 된다
- 버튼 크기와 형태는 브랜드 톤을 반영해야 한다

### 10.2 카드

- 아무 곳에나 카드부터 쓰지 않는다
- 카드가 필요한 이유가 있을 때만 사용한다
- 정보 구획, 선택 가능성, 반복 데이터 표현 같은 목적이 분명해야 한다

### 10.3 네비게이션

- 랜딩 페이지 헤더는 가볍게
- 복잡한 앱 헤더는 정보 구조를 명확하게
- 스티키 헤더는 맥락을 도와야지 시야를 빼앗으면 안 된다

### 10.4 폼

- 입력 흐름이 안정적이어야 한다
- 설명, 상태, 오류 피드백이 명확해야 한다
- 장식보다 사용성이 우선이다

### 10.5 이미지와 미디어

- 단순 장식이 아니라 메시지 전달에 기여해야 한다
- 텍스트와 경쟁하지 않게 한다
- 크롭, 비율, 오버레이 톤을 전체 컨셉과 맞춘다

---

## 11. 모션 규칙

모션은 “있으면 멋있다”가 아니라 **의도를 강화한다**가 기준이다.

좋은 모션:

- 페이지 로드 시 계층을 보여준다
- hover/focus 시 반응을 명확히 한다
- 스크롤 전환의 리듬을 만든다
- 너무 빠르거나 산만하지 않다

나쁜 모션:

- 모든 요소가 동시에 뜬다
- easing이 제각각이다
- hover마다 과한 확대/회전이 들어간다
- 읽는 흐름을 방해한다

권장:

- 진입 애니메이션은 몇 가지 패턴으로 통일
- `transform` / `opacity` 중심
- 인터랙션 밀도는 브랜드 톤에 맞춤

---

## 12. 접근성과 반응형

예쁜 디자인과 접근성은 충돌하지 않는다.

반드시 지킬 것:

- 충분한 명도 대비
- 키보드 포커스 표시
- 시맨틱 태그 사용
- 아이콘 버튼에 레이블 제공
- 모바일에서 터치 영역 충분히 확보
- 작은 화면에서 위계가 무너지지 않게 재배치

모바일 대응 시 주의:

- 데스크톱 레이아웃을 단순 축소하지 않는다
- 정보 순서를 다시 정렬할 수 있다
- 장식 효과는 줄이고 핵심 메시지를 앞당긴다

---

## 13. 구현 디테일

에이전트가 코드를 작성할 때 우선할 것:

- CSS 변수 사용
- spacing / radius / shadow 체계 정리
- 의미 있는 컴포넌트 분리
- 무거운 효과 남발 금지
- 배경, 노이즈, 그라디언트는 한 겹씩 이유 있게 추가

좋은 코드 특징:

- 테마 토큰이 정리되어 있다
- 스타일 수치가 우발적이지 않다
- 반복이 줄어 있다
- 반응형 규칙이 초기에 고려돼 있다

나쁜 코드 특징:

- 픽셀 값이 무질서하게 늘어난다
- 컴포넌트마다 radius/shadow가 다르다
- 애니메이션이 제각각이다
- 레이아웃이 오직 데스크톱 기준이다

---

## 14. 절대 피해야 하는 결과물

아래 중 하나라도 강하게 보이면 실패로 본다.

- 어디서나 본 듯한 SaaS 랜딩
- 보라색 그라디언트 남용
- 카드 3개 반복
- 설명 없는 큰 블러나 유리 효과
- 폰트 선택에 성격이 없음
- 섹션별 시각 언어가 따로 놈
- 화면 어디를 봐야 할지 모르는 위계
- 과한 애니메이션으로 산만함
- 모바일에서 구조가 무너짐
- 접근성이나 읽기 속도를 희생한 스타일링

---

## 15. 에이전트 출력 방식

웹 UI 제작 요청을 받으면, 에이전트는 머릿속으로 아래를 먼저 정한 뒤 구현한다.

### 15.1 먼저 정하는 것

- 디자인 컨셉 한 문장
- 핵심 사용자 감정 한 문장
- 레이아웃 전략 한 문장

### 15.2 구현 시 반드시 보여야 하는 것

- 컨셉이 드러나는 첫 화면
- 섹션마다 다른 역할
- 통일된 타이포 / 컬러 / 간격
- 신뢰 가능한 CTA 구조
- 자연스러운 반응형 대응

### 15.3 가능한 경우 포함할 것

- 커스텀 커서 또는 포인터 반응
- 섬세한 패럴랙스 또는 reveal
- 독특한 divider / border / texture
- 브랜드에 맞는 이미지 프레이밍
- 너무 뻔하지 않은 navigation 처리

단, 이런 요소들은 **컨셉을 강화할 때만** 사용한다.

---

## 16. 품질 체크리스트

결과물을 만든 뒤 스스로 아래를 점검한다.

- 첫 화면만 보고 무드가 바로 읽히는가
- 어느 섹션이 가장 중요한지 한눈에 보이는가
- 카드/그리드가 반복되며 지루해지지 않는가
- 폰트와 컬러가 같은 세계관에 있는가
- 버튼 위계가 분명한가
- 모바일에서도 설득력이 유지되는가
- 접근성 기본이 깨지지 않았는가
- “다른 아무 서비스에도 붙일 수 있는 디자인”처럼 보이지 않는가

하나라도 애매하면 다시 다듬는다.

## 17. 빠른 프롬프트 템플릿

사용자가 Cursor에게 함께 던질 수 있는 기본 템플릿:

```md
이 페이지를 만들어줘.

목적:
- [무엇을 위한 페이지인지]

대상 사용자:
- [누가 보는지]

원하는 인상:
- [예: 조용한 프리미엄, 에디토리얼, 실험적, 따뜻한 브랜드]

핵심 행동:
- [예: 예약, 문의, 가입, 제품 탐색]

피하고 싶은 것:
- 뻔한 SaaS 랜딩
- 카드 3개 반복
- 과한 보라 그라디언트
- 템플릿 같은 섹션 구성

요구사항:
- 반응형
- 접근성 기본 준수
- 실제 구현 가능한 코드
- 컨셉이 분명한 타이포/컬러/레이아웃
```

## 18. 최종 지시

웹 UI를 만들 때는 아래를 따른다.

- 먼저 강한 방향을 고른다
- 그 방향과 맞지 않는 선택은 버린다
- 레이아웃을 브랜드와 콘텐츠에 맞게 설계한다
- 디테일로 완성도를 올린다
- 흔한 템플릿 조합으로 도망가지 않는다
- 아름다움과 사용성을 동시에 챙긴다
- 실제 구현 가능한 결과물을 만든다

좋은 결과물은 화려한 화면이 아니라,  
**의도가 선명하고, 밀도가 높고, 반복적이지 않으며, 실제로 잘 작동하는 화면**이다.
